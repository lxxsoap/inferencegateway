
## 📋 题目：实现大模型推理网关（Inference Gateway）

### **🎯 题目目标**

实现一个面向大模型推理场景的 HTTP 网关，支持：
- **智能路由**
- **负载均衡**  
- **KV Cache 感知调度**

**关键挑战**：在"提升缓存命中率"和"防止单机过载"之间做平衡。

---

### **📖 场景说明**

在大模型推理场景中，KV Cache是一种关键的性能优化机制。相同前缀的请求路由到同一后端，可复用KV Cache并降低延迟；但过度聚合会导致热点与拥塞。

你需要实现一个**带策略权衡的调度器**，而不是单纯轮询。

---

### **⚙️ 简化假设**

1. 后端模型服务可使用 **Mock**
2. **Prefix 匹配**可用字符串前缀
3. **负载**可用并发请求数近似

---

### **✅ 功能要求（必做）**

#### **1. 网关基础能力**
- 实现 HTTP 服务器，监听指定端口
- 提供推理接口：`POST /infer`
- 请求格式：
```json
{
  "model": "model_name",
  "prompt": "user input",
  "session_id": "optional_session_id"
}
```

#### **2. 多后端管理**
- 支持配置多个后端推理服务实例
- 后端配置示例：
```yaml
backends:
  - id: "backend-1"
    address: "http://localhost:8081"
    max_concurrency: 10
  - id: "backend-2"
    address: "http://localhost:8082"
    max_concurrency: 10
```

#### **3. 智能路由策略**
实现以下路由策略（至少实现前2种）：

**策略1：Prefix Cache 感知路由**
- 相同前缀的请求应尽可能路由到同一后端
- 利用KV Cache复用提升推理效率
- 需要维护前缀到后端的映射关系

**策略2：负载均衡**
- 考虑后端当前负载（并发请求数）
- 避免将请求发送到已满的后端
- 防止单机过载

**策略3：混合策略（加分）**
- 在缓存命中率和负载均衡之间做权衡
- 例如：当首选后端负载过高时，选择次优后端

#### **4. 健康检查**
- 定期检查后端健康状态
- 自动剔除不健康的后端
- 后端恢复后自动重新加入

---

### **🌟 进阶要求（选做）**

#### **1. 动态权重调整**
- 根据后端响应时间动态调整权重
- 慢节点自动降权

#### **2. 请求排队与超时**
- 当所有后端都过载时，支持请求排队
- 实现请求超时控制

#### **3. 缓存淘汰策略**
- 实现LRU等缓存淘汰策略
- 管理Prefix Cache的内存使用

#### **4. 可观测性**
- 提供 metrics 接口（Prometheus格式）
- 监控指标包括：
  - 各后端请求数
  - 缓存命中率
  - 平均响应时间
  - 错误率

---

### **🛠️ 技术栈要求**

- **语言**：Go / Java / Python（推荐Go）
- **框架**：标准库或轻量级框架
- **并发模型**：Goroutine/线程池

---

### **💡 核心考察点**

1. **并发编程能力**：线程/协程安全、锁的使用
2. **算法设计**：路由策略、负载均衡算法
3. **系统设计**：模块化设计、扩展性
4. **工程实践**：错误处理、日志、配置管理
5. **AI场景理解**：KV Cache原理、推理优化

---

### **📝 示例场景**

```
请求1: prompt="请介绍一下北京" → 路由到 Backend-1
请求2: prompt="请介绍一下北京的天气" → 优先路由到 Backend-1（前缀相同，复用KV Cache）
请求3: prompt="请介绍一下上海" → 可路由到 Backend-2（新前缀）
请求4: prompt="请介绍一下北京的景点" → 如果Backend-1负载过高，可选择Backend-2
```
